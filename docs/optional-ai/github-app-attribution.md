# GitHub attribution for generated contributions

This maintainer-side workflow gives a private local assistant a distinct,
searchable GitHub identity without making that assistant part of WyrmGrid. The
assistant never receives a GitHub credential, never operates the repository,
and never gains review, merge, release, or version authority. A person can still
perform every contribution step manually.

Hoardmind is the current maintainer's private example. Another maintainer may
use their own assistant ID, display name, model, and GitHub App. The durable
branch form is `assistant/<assistant-id>/<contribution-id>`.

## Attribution policy

Use the GitHub App path only when the proposed patch itself was generated by the
named assistant and has not been materially rewritten by a person. The App
creates one commit and one identity-bound branch. After the App token is
discarded, the human maintainer's authenticated GitHub CLI creates the draft
pull request. The commit omits custom Git author and committer fields so GitHub
can attribute and, where supported, verify it as the App bot.

When a person implements or materially rewrites an assistant's suggestion, that
person remains the commit author. Record narrow assistance in the commit or pull
request body with `Assisted-by: <assistant name>` instead of publishing it as a
bot-authored change.

One generated task should remain one pull request and one squash merge. Preserve
the `Optional-AI-Contribution`, input-packet hash, output-patch hash, and human-
review trailers in the final squash message. This makes the main-branch change
one explicit revert unit while the original App commit and draft review record
remain attached to the pull request.

## One-time GitHub App registration

Register a private GitHub App owned by the maintainer account or organization.
For the current private assistant, the recommended app name and slug are
`WyrmGrid Hoardmind` and `wyrmgrid-hoardmind`.

Grant only these repository permissions:

- **Contents: Read and write** — create one commit and branch;
- **Metadata: Read-only** — GitHub supplies this implicitly.

Do not grant Actions, Administration, Checks, Codespaces, Deployments,
Environments, Issues, Members, Secrets, Workflows, organization, or user
permissions. Disable webhooks unless a separately reviewed use case needs them.
Install the App on `phobos-dthorga/onair-wyrmgrid` only, not every repository.

Protect `main` with a repository ruleset that requires changes to arrive through
a pull request, and do not add the App as a bypass actor. Contents permission is
necessarily capable of writing Git references; the ruleset prevents a stolen
App credential from writing directly to the protected branch. The App has no
Pull requests permission, so it cannot open, review, approve, or merge a PR.

Generate one private key in the App settings. Store it outside this repository,
for example under a maintainer-only directory in the operating-system local
application-data area. Restrict its filesystem permissions to the maintainer.
Never paste the key into a prompt, chat, packet, metrics file, issue, pull
request, shell argument, environment variable, or repository file.

Copy
[`examples/optional-ai/github-app-installation-config-v1.json`](../../examples/optional-ai/github-app-installation-config-v1.json)
to `.wyrmgrid-local/github-app.json`, then replace the example App ID,
installation ID, slug, and external private-key path. The local directory is
ignored by Git. App and installation IDs are identifiers rather than
credentials, but the configuration stays local because its key path and
repository authority are maintainer-specific.

## Generated-patch workflow

The built-in `implementation-patch-v1` task drafts a narrow patch from a
sanitized, reviewer-selected packet. It remains a no-tools, review-only model
call. Run it with `npm run ai:implementation-patch` and retain its draft and
metrics outside the repository.

Extract the one allowed fenced diff into a private patch file:

```powershell
npm run ai:contribution:extract -- `
  --draft <generated-draft.md> `
  --output .wyrmgrid-local\hm-20260718-001.patch `
  --approve-once
```

Review the patch, then bind it to the exact input packet, model identity,
allowed paths, branch, and optional metrics file:

```powershell
npm run ai:contribution:prepare -- `
  --patch .wyrmgrid-local\hm-20260718-001.patch `
  --packet <sanitized-packet.md> `
  --metrics <privacy-reduced-metrics.json> `
  --output .wyrmgrid-local\hm-20260718-001.manifest.json `
  --contribution-id hm-20260718-001 `
  --assistant-id hoardmind `
  --assistant-name Hoardmind `
  --model qwen3-coder:30b `
  --task-contract implementation-patch-v1 `
  --branch assistant/hoardmind/hm-20260718-001 `
  --commit-subject "test: add generated boundary coverage" `
  --pr-title "Add generated boundary coverage" `
  --summary "Adds one generated regression case for maintainer review." `
  --allow apps/desktop/src/lib/example.test.ts `
  --approve-once
```

The preparation result prints the manifest SHA-256. Review the generated
manifest and copy that digest into the separate publish invocation:

```powershell
npm run ai:contribution:publish -- `
  --manifest .wyrmgrid-local\hm-20260718-001.manifest.json `
  --patch .wyrmgrid-local\hm-20260718-001.patch `
  --config .wyrmgrid-local\github-app.json `
  --expected-manifest-sha256 <reviewed-manifest-sha256> `
  --approve-once
```

The approval applies to that invocation only. Publication uses a short-lived
Contents-only installation token for the bot commit and branch, discards it,
then uses the maintainer's existing GitHub CLI session to open a draft PR. It
does not merge, mark ready, release, tag, change a version, start or rerun hosted
checks, or send the patch back to a model.

## Provenance-preserving squash landing

After reviewing the draft, running the applicable local gates, updating any
protected changelog or policy files under human authorship, and marking the PR
ready, preview the exact landing evidence and squash message:

```powershell
npm run ai:contribution:landing-preview -- `
  --manifest .wyrmgrid-local\hm-20260718-001.manifest.json `
  --config .wyrmgrid-local\github-app.json `
  --expected-manifest-sha256 <reviewed-manifest-sha256> `
  --expected-head-sha <exact-bot-commit-sha> `
  --pr <pull-request-number>
```

The preview requires the authenticated GitHub CLI actor to be a person and
checks the repository, `main` base, identity-bound branch, exact single commit,
App-bot identity, original bot commit message, and clean protected merge state.
Review the displayed subject and body, then give a separate one-invocation
landing approval:

```powershell
npm run ai:contribution:land -- `
  --manifest .wyrmgrid-local\hm-20260718-001.manifest.json `
  --config .wyrmgrid-local\github-app.json `
  --expected-manifest-sha256 <reviewed-manifest-sha256> `
  --expected-head-sha <exact-bot-commit-sha> `
  --pr <pull-request-number> `
  --approve-once
```

The guard supplies the complete squash message rather than accepting GitHub's
default, binds the merge to the exact bot head, never uses an administrative
bypass, and reads the resulting merge commit back from GitHub to verify every
provenance trailer. The App remains unable to review or merge; this command uses
the human maintainer's GitHub CLI identity.

## Fail-closed broker controls

`scripts/optional-ai-contribution.mjs` rejects:

- CI execution, missing one-time approval, dirty local worktrees, stale base
  commits, branch reuse, changed manifest or patch hashes, and App-slug mismatch;
- binary patches, oversized content, path traversal, quoted or space-containing
  paths, deletions, renames, copies, file-mode changes, submodules, and
  executable generated files;
- changes outside reviewer-approved exact paths or prefixes;
- workflows and other `.github` policy, contributor instructions, dependency
  manifests, migrations, legal/security policy, protocol/schema definitions,
  release automation, or the optional-AI governance and broker itself;
- private keys and common high-confidence credential patterns in the patch or
  resulting files.

`scripts/optional-ai-landing.mjs` independently rejects a changed manifest,
non-human GitHub actor, wrong repository/base/branch/head, multiple commits,
spoofed App identity, altered bot message, draft or unclean PR, missing fresh
approval, and any merged squash message that loses provenance.

Sanitized schema fixtures remain eligible, but schema definitions remain
protected. A blocked patch is not automatically weakened or retried: a person
must either narrow it or implement the change under their own authorship.

## Review and reversal

Treat the generated PR exactly like any other untrusted contribution. Reconcile
the patch against source evidence, run all applicable local quality gates,
update compatibility and changelog records under human authorship when those
protected decisions are required, and explicitly decide whether to land it.

To reverse a landed generated contribution, revert its single squash commit and
reference the contribution ID and original pull request. Attribution helps find
the change; the commit identity—not the assistant account—is what makes the
reversal deterministic.

If GitHub accepts the bot branch but the maintainer's draft-PR creation fails,
do not rerun publication against that branch. Inspect the published commit,
then either open the draft PR manually with the maintainer's GitHub CLI session
or remove the unmerged branch under the maintainer identity and prepare a fresh
contribution ID. The broker exposes neither branch deletion nor PR repair to the
App.
